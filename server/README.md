# Server-side configuration

Reference copies of the nginx and cron configuration that live on the EC2
instance. **These files are NOT auto-deployed** — `./deploy.sh` only
deploys the Next.js app. Changes here have to be `scp`'d to EC2 and
applied manually. See "Applying changes" below.

## Why these aren't auto-deployed

The Next.js app gets redeployed often (several times a week). The
nginx + cron layer changes maybe once a month or less, and a bad
nginx config can take the whole site down in a way that's harder
to auto-rollback than a bad app build. Keeping them out of the
automated deploy path means:

- A bad commit to the app can't accidentally take nginx down
- Nginx changes happen with conscious intent, by a human, with
  `nginx -t` validation before reload
- The repo still has authoritative copies so the box is
  reproducible if EC2 gets rebuilt

The trade-off: it is **possible for the repo and the live server
to drift**. If you change a file here, you must push to EC2 yourself.
If someone (Carl, a future Claude session, Jesse) makes a change on
the EC2 box without updating the repo, the repo gets stale. The
runbook below covers how to stay in sync.

## What's here

| File | Lives on EC2 at | Purpose |
|---|---|---|
| `nginx/00-bot-block.conf` | `/etc/nginx/conf.d/00-bot-block.conf` | User-agent map: returns `$blocked_ua=1` for AI training crawlers and heavy SEO scrapers (Meta, OpenAI, Anthropic, DataForSeo, Semrush, etc.) |
| `nginx/holylandphotos.conf` | `/etc/nginx/conf.d/holylandphotos.conf` | Main vhost. Serves apex on 443, redirects www→apex, redirects HTTP→HTTPS. Has `if ($blocked_ua) { return 403; }` and `if ($blocked_country) { return 403; }` lines. |
| `nginx/default.conf` | `/etc/nginx/conf.d/default.conf` | Catch-all default server for HTTP requests with unknown hostnames or direct-IP access. |
| `nginx/refresh-country-blocklist.sh` | `/usr/local/sbin/refresh-country-blocklist` | Regenerates `/etc/nginx/conf.d/01-country-block.conf` from upstream IP lists (ipdeny.com + RADB). Invoked by the systemd timer below. Safe to run by hand as root. |
| `systemd/hlp-blocklist-refresh.service` | `/etc/systemd/system/hlp-blocklist-refresh.service` | Oneshot service that runs the refresh script. |
| `systemd/hlp-blocklist-refresh.timer` | `/etc/systemd/system/hlp-blocklist-refresh.timer` | Weekly timer (Mon 03:30 UTC + random delay). `Persistent=true` so a missed run fires on next boot. Matches the pattern of `hlp-backup.timer`. |

**NOT committed:**

- `/etc/nginx/conf.d/01-country-block.conf` — auto-generated by the
  refresh script, ~18k lines of CIDR entries. Regenerates on its own
  weekly. No value to versioning.
- `/etc/nginx/blocklists/*.zone` — the raw upstream IP lists, also
  auto-refreshed.
- `/etc/letsencrypt/**` — TLS certs, managed by `certbot`.

## Architecture: how bots get blocked

Three layers, complementary:

1. **`robots.txt`** ([`src/app/robots.ts`](../src/app/robots.ts)) —
   asks polite bots not to crawl. Honoured by Meta, Google AI,
   Anthropic, OpenAI within 24-48h. Has no effect on bots that
   ignore the standard.

2. **User-agent blocklist** (`00-bot-block.conf`) — nginx drops
   requests with named bot user agents at the edge (HTTP 403),
   before they hit the Next.js app. Effective immediately. Real
   search bots (Googlebot, bingbot, DuckDuckBot, Applebot) are
   deliberately NOT blocked here so the site stays indexable.

3. **Geo/ASN blocklist** (`01-country-block.conf`, regenerated by
   `refresh-country-blocklist.sh`) — nginx drops requests from IPs
   in **Singapore, China, or Vietnam** CIDR ranges, or in
   Byteplus/Tencent/Zenlayer ASN ranges. Catches bots that disguise
   their user agent as a normal browser. Re-run weekly via the
   `hlp-blocklist-refresh.timer` systemd timer so the CIDR data
   stays current.

   Why these countries: in the GA4 + nginx access log analysis,
   each had 0% engagement, 0s average engagement time, and a request
   pattern dominated by bots — Baidu/Sogou + scrapers (CN), residential
   proxy networks (VN), and crawlers/scrapers (SG). No real audience
   for an English-language scholarly site lives behind those IPs.

Order of evaluation per request:
```
client connect → nginx
                  ↓
       check $blocked_ua  (UA map from 00-bot-block.conf)
                  ↓ no
       check $blocked_country  (geo map from 01-country-block.conf)
                  ↓ no
       proxy_pass to Next.js on 127.0.0.1:3000
                  ↓
       Next.js + Payload + DB respond
```

If either check matches, nginx returns 403 immediately — Next.js never
sees the request, the DB is untouched, GA4 is not fired. That last
point matters: bots blocked at nginx don't pollute analytics.

## Runbook

### Add a new bot user agent to the blocklist

Most common change. Edit [`nginx/00-bot-block.conf`](nginx/00-bot-block.conf):
add a new line under the `map $http_user_agent $blocked_ua` block
like `"~*MyNewBot"  1;`.

Then sync + reload:
```bash
scp -i ~/.ssh/hlp-ec2-key.pem server/nginx/00-bot-block.conf \
  ec2-user@18.220.101.13:/tmp/00-bot-block.conf
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo mv /tmp/00-bot-block.conf /etc/nginx/conf.d/00-bot-block.conf \
   && sudo nginx -t && sudo systemctl reload nginx'
```

Confirm with `curl`:
```bash
curl -sI -A 'MyNewBot/1.0' https://holylandphotos.org/ | head -1
# → HTTP/2 403
```

Also add the bot to the [`robots.ts`](../src/app/robots.ts) disallow
list so polite-but-ignored-by-us bots have a chance to back off
voluntarily.

### Add a new country to block

Edit the `COUNTRIES=(…)` array near the top of
`refresh-country-blocklist.sh`. For example, to add Russia:
```bash
COUNTRIES=(
    "sg:/etc/nginx/blocklists/sg.zone"
    "cn:/etc/nginx/blocklists/cn.zone"
    "vn:/etc/nginx/blocklists/vn.zone"
    "ru:/etc/nginx/blocklists/ru.zone"   # new
)
```
The code (`ru`) must match the slug ipdeny.com uses in
`https://www.ipdeny.com/ipblocks/data/aggregated/<code>-aggregated.zone`.
Then sync + run once manually (see below).

### Add a new bot-hosting ASN to block

Edit the `for asn in 150436 132203 4229; do` line in
`refresh-country-blocklist.sh` to include the new ASN number.
Then sync + run once manually.

### Run the refresh manually (e.g. after editing the script)

```bash
scp -i ~/.ssh/hlp-ec2-key.pem server/nginx/refresh-country-blocklist.sh \
  ec2-user@18.220.101.13:/tmp/refresh-country-blocklist
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo install -m 0755 -o root -g root /tmp/refresh-country-blocklist /usr/local/sbin/refresh-country-blocklist \
   && sudo systemctl start hlp-blocklist-refresh.service \
   && journalctl -u hlp-blocklist-refresh.service -n 30 --no-pager'
```

The service is `Type=oneshot`, so `systemctl start` blocks until the
script finishes. It downloads fresh CIDR lists, regenerates
`01-country-block.conf`, runs `nginx -t`, and reloads. If `nginx -t`
fails it does NOT reload — the previous config keeps serving.

### Verify the weekly timer is healthy

```bash
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'systemctl list-timers hlp-blocklist-refresh.timer --no-pager; \
   sudo ls -la /etc/nginx/blocklists/'
```

What to look for:
- `list-timers` should show a `NEXT` time within the next 7 days and
  a `LAST` time within the past ~7 days (the first scheduled run won't
  populate `LAST` until it fires).
- All `.zone` file modification timestamps should be within the past
  ~7 days. If they're older than 8–9 days, the timer isn't firing —
  check `systemctl status hlp-blocklist-refresh.timer` and
  `journalctl -u hlp-blocklist-refresh.service`.

### Debug "is this IP/UA being blocked?"

```bash
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo tail -10000 /var/log/nginx/access.log | grep <IP-OR-UA-SUBSTRING>'
```
The status field (9th column) shows 403 for blocked, 200 for allowed.

### Roll back if a change took the site down

The previous version of each vhost file is kept on EC2 as
`*.bak-<timestamp>` (created by our own scripts when we rewrite a
file). Roll back with:
```bash
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo ls /etc/nginx/conf.d/*.bak* | sort | tail -1'   # find the latest
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo cp /etc/nginx/conf.d/holylandphotos.conf.bak-XXX /etc/nginx/conf.d/holylandphotos.conf \
   && sudo nginx -t && sudo systemctl reload nginx'
```

### Pull the current EC2 state back into the repo

If you (or someone) made a change directly on the box and want the
repo to reflect reality:
```bash
scp -i ~/.ssh/hlp-ec2-key.pem \
  ec2-user@18.220.101.13:/etc/nginx/conf.d/00-bot-block.conf \
  ec2-user@18.220.101.13:/etc/nginx/conf.d/holylandphotos.conf \
  ec2-user@18.220.101.13:/etc/nginx/conf.d/default.conf \
  server/nginx/
ssh -i ~/.ssh/hlp-ec2-key.pem ec2-user@18.220.101.13 \
  'sudo cat /usr/local/sbin/refresh-country-blocklist' \
  > server/nginx/refresh-country-blocklist.sh
chmod +x server/nginx/refresh-country-blocklist.sh
scp -i ~/.ssh/hlp-ec2-key.pem \
  ec2-user@18.220.101.13:/etc/systemd/system/hlp-blocklist-refresh.service \
  ec2-user@18.220.101.13:/etc/systemd/system/hlp-blocklist-refresh.timer \
  server/systemd/
git diff server/   # review
```

## Bootstrapping a fresh EC2 box from this repo

If you ever rebuild the EC2 instance from scratch, after the basic
nginx install + Let's Encrypt cert, the blocking layer is restored by:

```bash
# whois is used by the refresh script for RADB ASN lookups
sudo dnf install -y whois

# nginx configs
sudo install -m 0644 -o root -g root server/nginx/00-bot-block.conf /etc/nginx/conf.d/
sudo install -m 0644 -o root -g root server/nginx/holylandphotos.conf /etc/nginx/conf.d/
sudo install -m 0644 -o root -g root server/nginx/default.conf /etc/nginx/conf.d/

# Refresh script + systemd units
sudo install -m 0755 -o root -g root \
  server/nginx/refresh-country-blocklist.sh /usr/local/sbin/refresh-country-blocklist
sudo install -m 0644 -o root -g root \
  server/systemd/hlp-blocklist-refresh.service /etc/systemd/system/
sudo install -m 0644 -o root -g root \
  server/systemd/hlp-blocklist-refresh.timer /etc/systemd/system/

# Enable + start the weekly timer
sudo systemctl daemon-reload
sudo systemctl enable --now hlp-blocklist-refresh.timer

# Populate the blocklists for the first time (the timer would do this
# next Monday otherwise; this gets you protected immediately)
sudo systemctl start hlp-blocklist-refresh.service
```

Note: do NOT install the refresh script into `/etc/cron.weekly/`. AL2023
ships without `anacron` and with `crond` inactive by default, so anything
in `/etc/cron.weekly/` is silently never run. The systemd timer above is
what actually fires.
